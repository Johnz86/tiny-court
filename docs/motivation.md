# Document 2: Agent Motivation and Focus Brief

## Mission

You are not building the biggest AI system.

You are building the sharpest small AI experience.

The hackathon rewards projects that are focused, polished, memorable, and honest about what small models can do well. Your job is to create an application that feels complete, useful, delightful, and easy to understand in the first minute.

The model limit is not a weakness. It is the creative constraint.

Build small. Make it sing.

---

# Primary Objective

Create a Gradio app on Hugging Face Spaces that uses models under the 32B parameter limit and delivers a clear user experience.

The app should succeed in one of two ways:

1. It solves a real problem for a real person.
2. It creates a delightful AI experience that users want to share.

Do not build a generic chatbot.

Do not build a vague assistant.

Do not build a feature list.

Build a tiny, complete experience with a strong point of view.

---

# If Competing in Backyard AI

## Core Focus

Focus on one real person and one real pain.

The best Backyard AI project should be explainable like this:

“I know this person. They have this repeated problem. I built this app. They used it. It helped in this measurable way.”

## What the Agent Should Prioritize

### 1. Specificity

The app should not serve “everyone.”

It should serve one clearly defined user.

Examples:

* a café owner making staff schedules,
* a teacher preparing reading materials,
* a parent organizing medical paperwork,
* a local club organizer managing event messages,
* a freelancer sorting client requests,
* a renter understanding confusing documents.

Specific beats impressive.

### 2. Real Usage

The app should be tested by the intended person.

Collect:

* a quote,
* a screenshot,
* a before/after comparison,
* one measurable improvement,
* one change made after feedback.

Even a tiny piece of real user evidence is powerful.

### 3. Honest Small-Model Fit

The app should use AI where small models are naturally strong:

* summarizing,
* extracting fields,
* classifying messages,
* rewriting text,
* structuring messy input,
* generating checklists,
* drafting replies,
* turning unstructured notes into useful outputs.

Avoid tasks that need massive world knowledge, high-stakes reasoning, or unreliable long-chain judgment.

### 4. Polished Workflow

The user should not need to think about prompts.

The app should guide them.

Good flow:

* paste messy input,
* click one clear button,
* review structured result,
* edit or confirm,
* export or share.

### 5. Measurable Better Day

The app should make the user’s day measurably better.

Possible measures:

* minutes saved,
* fewer errors,
* less confusion,
* clearer next action,
* faster communication,
* reduced stress.

## Backyard AI Winning Standard

The project should make a judge think:

“This is small, real, and useful. I believe someone actually needed this.”

# If Competing in Thousand Token Wood

## Core Focus

Focus on delight.

The best Thousand Token Wood project should be explainable like this:

“I tried it, it surprised me, I smiled, and I wanted to show someone else.”

## What the Agent Should Prioritize

### 1. Originality

Do not make another generic story generator.

Create a specific frame.

Examples:

* a tiny courtroom for everyday crimes,
* a bureau for lost things,
* a museum of alternate histories,
* a dream cartography office,
* a complaint choir,
* a bureaucratic goblin permit desk,
* a weather station for emotions.

The concept should be memorable in one sentence.

### 2. AI as the Fun Engine

The AI must do the central creative work.

It should generate:

* strange witnesses,
* personalized case files,
* evolving story logic,
* unexpected artifacts,
* fake bureaucratic documents,
* surreal rules,
* interactive consequences,
* adaptive dialogue.

If the AI disappeared, the app should lose its magic.

### 3. Interaction, Not Just Output

The app should not be one big text generation.

The user should make choices.

Good interactions:

* object,
* cross-examine,
* submit evidence,
* call witness,
* open next door,
* inspect artifact,
* file appeal,
* add twist,
* choose route,
* unlock ending.

The user should feel like they are playing, not reading.

### 4. Short Loops

The first successful experience should happen quickly.

The user should get a satisfying result in 60–90 seconds.

A longer mode can exist, but the first run must be fast.

### 5. Shareable Ending

The app should produce a final artifact.

Examples:

* court verdict card,
* lost object case file,
* museum plaque,
* dream map,
* creature passport,
* official certificate,
* tiny newspaper clipping.

The result should be screenshot-worthy.

## Thousand Token Wood Winning Standard

The project should make a judge think:

“This is weird, delightful, polished, and only works because AI is inside it.”

# Cross-Track Priorities

Regardless of track, focus on these.

## 1. Make the First 10 Seconds Clear

The landing page should answer:

* What is this?
* What do I type?
* What happens when I click?
* Why is it fun or useful?

If the user needs instructions, the design is too complex.

## 2. Avoid Generic Chat

A chat box alone is not enough.

The app should have:

* named actions,
* structured outputs,
* state,
* buttons,
* cards,
* progress,
* a final result.

The user should feel guided.

## 3. Make the Gradio App Look Custom

Visual polish matters.

Push beyond the default look with:

* themed layout,
* clear panels,
* strong title,
* cards,
* progress indicators,
* custom CSS,
* final share card,
* expressive labels.

A memorable interface can compete for the Off-Brand Award.

## 4. Make the Demo Video Excellent

The demo video is part of the submission.

The first 15 seconds should show the app’s value.

Good demo structure:

1. Show the problem or premise.
2. Enter one strong example.
3. Click the main action.
4. Show the surprising/useful result.
5. Show one interaction.
6. Show the final artifact.
7. End with the one-sentence pitch.

Do not spend the demo explaining the model.

Show the experience.

## 5. Optimize for Judge Memory

Judges will see many submissions.

Make the project easy to remember.

A good project has a phrase like:

* “AI court for petty crimes”
* “shift scheduler from chaotic staff messages”
* “lost-and-found bureau for surreal objects”
* “lease explainer for one real renter”
* “repair triage for one local bike shop”

If the judge cannot describe the app after one minute, it is too vague.

# Bonus Quest Strategy

The agent should not chase every bonus quest at the cost of the main experience.

The best priority order is:

## Highest Value

### Off-Brand

A strong custom UI helps both tracks and can win a special award.

### Best Demo

A great demo improves the main judging and can win a special award.

### Field Notes

A blog post or report is achievable and helps explain the build story.

## Medium Value

### Sharing is Caring

Useful if the project is agentic and has interesting traces.

### Tiny Titan

Very valuable if the app works well on a very small model, but do not sacrifice quality unless the small model genuinely fits.

## Conditional Value

### Off the Grid

Great if feasible, but risky if it hurts reliability.

### Well-Tuned

Useful if fine-tuning clearly improves the app, but not necessary for a strong submission.

### Llama Champion

Good if aligned with the technical plan, but should not dominate the product design.

# Recommended Focus for Tiny Court of Everyday Crimes

If building Tiny Court of Everyday Crimes for Thousand Token Wood, focus on the following.

## One-Sentence Pitch

Tiny Court of Everyday Crimes turns petty daily annoyances into dramatic interactive comedy trials with charges, evidence, objections, witnesses, verdicts, and harmless ridiculous sentences.

## What Must Be Excellent

### 1. The First Case

The first generated case must be funny immediately.

Use a strong default example:

“My cat knocked over my coffee and looked proud.”

### 2. The Interaction Buttons

The app must feel like a mini-game.

Essential buttons:

* Open Court
* Submit Evidence
* Call Witness
* Object!
* Cross-Examine
* Add Twist
* Deliver Verdict
* Appeal
* Generate Court Record

### 3. The Court Record

The final output should be a shareable artifact.

It should include:

* case title,
* charge,
* verdict,
* sentence,
* best quote,
* fake court seal.

### 4. The UI

The interface should look like a tiny court:

* docket cards,
* evidence cards,
* role-based transcript cards,
* verdict stamp,
* courtroom dignity meter,
* petty severity meter.

This gives the project a chance at the Off-Brand Award.

### 5. The Demo

The demo should be under two minutes.

Show:

1. user enters petty crime,
2. court opens,
3. prosecution and defense argue,
4. user clicks Object!,
5. user submits evidence,
6. verdict changes,
7. final court record appears.

## What to Avoid

Do not make it only a chatbot.

Do not generate long walls of text.

Do not use serious legal framing.

Do not handle real serious disputes.

Do not let the user wait too long before the first funny moment.

## Winning Standard for Tiny Court

The app wins if judges laugh, interact, and remember it as:

“The tiny AI courtroom for stupid household crimes.”

---

# Final Motivation

Build something that feels complete.

Small does not mean weak.

Small means focused.

A winning project does not need to solve everything. It needs to solve one thing beautifully or create one delightful experience completely.

The model is small.

The app should be sharp.

The demo should be clear.

The interaction should be memorable.

The final artifact should be shareable.

If a judge smiles, understands the concept, clicks another button, and wants to show someone else, the project is on the right path.
